# ТЗ: dosfm — браузерная DOS-консоль для управления файлами (js-dos + FastAPI)

## Цель
Веб-приложение: в браузере открывается **DOS-консоль (js-dos = DOSBox в WebAssembly)**, где запускается
**DOS-файловый менеджер** (типа Volkov Commander / DOS Navigator) для **управления файлами**.
Упор — на управление файлами; запуск DOS-игр — приятный бонус.

## Согласованный scope
- **Бэкенд:** Python + FastAPI (наш стандартный стек).
- **js-dos:** центральная роль — реальный запуск DOS в браузере (не декорация).
- **Файлы:** **серверная персистентная рабочая папка**. Файлы лежат на бэкенде; DOS-менеджер в браузере
  их редактирует; изменения **синхронизируются обратно на сервер**.
- **Объём:** минимум (один пользователь/админ, одна рабочая папка, без ролей/тем/медиатеки).

## Архитектура (ключевое)
- Бэкенд хранит файлы в рабочей папке `workspace/`. API:
  - `GET /api/tree` — дерево файлов рабочей папки;
  - `GET /api/download?path=` — отдать файл;
  - `POST /api/sync` — принять изменённые файлы из эмулятора и записать в `workspace/` (безопасно, без выхода за пределы).
- Фронт: грузит js-dos, поднимает бандл с DOS-файловым менеджером, заполняет диск файлами с сервера
  (`fsWriteFile`), даёт кнопку/авто «Синхронизировать» → читает ФС эмулятора (`fsTree`/`fsReadFile`) и шлёт на `/api/sync`.
- Мост js-dos↔сервер (заливка на старте + выгрузка изменений) — центральная часть.

## Критерии готовности (по вехам — см. docs/project-notes.md)
- Бэк: API рабочей папки (`tree`/`download`/`sync`), защита от path traversal — тесты зелёные.
- Фронт: js-dos поднимается, DOS-менеджер видит файлы с сервера; правки сохраняются на сервер (проверка руками).
- `/security` пройден; записано в docs + осмысленный коммит.

## Открытые вопросы / решить по ходу
1. **Какой DOS-менеджер бандлить** — с учётом лицензий (для себя, но лучше свободный): DOS Navigator (open source, BSD)
   или FreeDOS-инструменты. Volkov Commander — freeware, но не открытый. По умолчанию целюсь в свободный/открытый.
2. js-dos: версия и способ подключения (CDN vs локальный vendor). По умолчанию — локальный vendor (офлайн, стабильно).
3. Аутентификация: в «минимуме» — без входа (или простой токен). Если публично — добавить простой гейт (спросить ПМ).

## Явно НЕ входит
- Многопользовательность, роли, темы, медиатека.
- Онлайн-мультиплеер игр, запись/шаринг сессий.

## Безопасность (заранее)
- `POST /api/sync` и `download` — строго в пределах `workspace/` (никакого `..`/абсолютных путей). Это красная зона (запись файлов) — валидация путей обязательна, ревью ПМ по auth, если появится.
